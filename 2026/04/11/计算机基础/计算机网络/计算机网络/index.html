TCP 如何实现可靠传输和流量控制？

（1）如何实现可靠传输？

TCP 的可靠传输不是靠单一机制完成的，而是一套“组合拳”，确保数据无差错、不丢失、不重复、且按序到达：

  1. 校验和（Checksum）： 检查数据在传输过程中是否发生了物理损坏（位错误），如果校验失败直接丢弃。
  2. 序列号与确认应答（Seq & ACK）：
    • 防乱序与去重： TCP 给每个字节都打上序号（Sequence Number），接收方可以根据序号重新排序，并丢弃重复收到的数据。
    • 确认机制： 接收方收到数据后，必须回传一个 ACK（确认号），告诉发送方“我收到了”。
  3. 重传机制： （如上题所述）通过超时重传和快速重传，解决报文在网络中丢失的问题。
  4. 连接管理： 通过“三次握手”建立可靠连接，通过“四次挥手”安全释放连接，确保双方都准备好收发数据。

（2）如何实现流量控制（Flow Control）？

核心目的： 防止“发送方发得太快”，导致“接收方处理不过来”从而撑爆接收方的缓存（针对的是端到端的接收能力）。
实现机制：滑动窗口（Sliding Window）

  1. 接收方在给发送方回传 ACK 报文时，会在 TCP 头部填入一个叫 rwnd（接收窗口，Receive Window） 的值。
  2. 这个值代表接收方目前还有多少空闲的缓存区可以接收数据。
  3. 发送方收到这个值后，会严格按照 rwnd 的大小来限制自己下一次发送的数据量。如果 rwnd=0，发送方就会停止发送，并启动“零窗口探测计时器”（如上一题所述）。

TCP 发送窗口如何根据接收窗口和拥塞情况动态调整，窗口结构如何维护变化？

（1）发送窗口的动态调整机制

发送方的**发送窗口（swnd, Send Window）**并不是单方面决定的，它受到“接收方处理能力”和“整个网络拥堵情况”的双重制约。
核心公式：发送窗口 swnd = min(接收窗口 rwnd, 拥塞窗口 cwnd)

  • 接收窗口（rwnd）： 来自接收方，反映接收方的缓存余量（流量控制）。
  • 拥塞窗口（cwnd，Congestion Window）： 是发送方自己维护的一个状态变量，反映当前网络的拥堵程度（拥塞控制）。
    • 慢启动（Slow Start）： 连接刚建立时，cwnd 从小开始，呈指数级增长。
    • 拥塞避免（Congestion Avoidance）： 当 cwnd 达到某个阈值时，改为线性增长，避免突然拥塞网络。
    • 网络拥塞发生时（丢包）： 如果发生超时重传，cwnd 瞬间降到极小值（如 1），重新慢启动；如果发生快速重传，cwnd 减半，进入“快速恢复”阶段。

结论： 发送方会取 rwnd 和 cwnd 中的较小值作为实际的发送窗口大小。谁的能力弱（接收方来不及，或网络塞车），就听谁的。

（2）窗口结构如何维护变化？

发送方的滑动窗口将数据分为四部分：

  1. 已发送且已确认（窗口左侧，不再关心）
  2. 已发送但未确认（占用窗口空间，随时准备重发）
  3. 允许发送但尚未发送（窗口内剩余的可用空间）
  4. 不允许发送（窗口右侧，必须等窗口向前滑动才能发送）

窗口的变化（滑动）：
当发送方收到接收方的 ACK 确认后，窗口的左边缘会向右移动（把“已确认”的数据移出窗口）。同时，根据接收方最新通告的 rwnd 大小，窗口的右边缘可能向右扩展（接收方处理完了旧数据，腾出了新空间），从而允许发送更多属于第 4 部分的数据。

#为什么 HTTP/3 基于 QUIC，而不继续在 TCP 上优化？

  1. TCP 的“队头阻塞”问题无法彻底根除（最核心原因）：
    • HTTP/2 引入了“多路复用”，在一个 TCP 连接上可以并发跑多个 HTTP 请求。
    • 但是，TCP 的底层设计是保证数据的“严格有序”。如果在同一个 TCP 连接中，某个数据包丢了，TCP 会暂停把后续收到的所有数据交给应用层，直到丢失的包重传成功。
    • 这意味着，只要丢一个包，这个 TCP 连接上的所有 HTTP/2 请求全部卡死！这被称为 TCP 层的队头阻塞。
  2. 建连延迟太高（握手太慢）：
    • TCP 三次握手需要 1 个 RTT（往返时间）。如果加上 HTTPS 的 TLS 加密握手，又需要 1~2 个 RTT。用户打开网页的初始延迟很高。
    • QUIC 将传输层握手和加密层握手合二为一，可以实现 1-RTT 甚至 0-RTT（首次连接过以后，再次连接无需握手直接发数据）。
  3. 网络迁移体验差：
    • TCP 连接由“四元组”（源 IP、源端口、目的 IP、目的端口）标识。当你拿着手机从 4G 切换到 Wi-Fi 时，IP 地址变了，TCP 连接就会断开，必须重新建立连接（导致视频卡顿、游戏掉线）。
    • QUIC 使用一个 64 位的 Connection ID（连接标识符） 来识别连接，只要 ID 没变，即使 IP 变了，连接依然无缝继续。

HTTP 队头阻塞是什么？

队头阻塞（Head-of-Line Blocking, HoL），顾名思义，就像十字路口等红灯，排在第一辆的车（队头）如果出了故障不动，后面的车即使要去不同的方向，也被堵死在后面无法前行。

在 HTTP 的演进中，队头阻塞发生过两次“变异”：

1. HTTP/1.1 时代的“应用层”队头阻塞

  • 原因： HTTP/1.1 规定，在同一个 TCP 连接上，请求可以连续发（Pipelining），但服务器必须严格按照请求的顺序返回响应。
  • 表现： 浏览器发了 A、B 两个请求。如果 A 是一个需要服务器处理很久的复杂请求（或者 A 的响应数据极大），那么哪怕 B 只是一张很小的图片且早处理完了，B 也必须等 A 完全发送完毕后才能返回。
  • 当时的解决办法： 浏览器干脆对同一个域名同时开启 6 个 TCP 物理连接，绕过限制。

2. HTTP/2 时代的“传输层 (TCP)”队头阻塞

  • 改进： HTTP/2 引入了二进制分帧和多路复用，A 和 B 两个请求的数据可以切碎了混合在同一个 TCP 连接里发送，谁先处理完谁先发，彻底解决了 HTTP 层的队头阻塞。
  • 新问题（TCP 队头阻塞）： 如上文所述，所有的请求现在都挤在唯一的 1 个 TCP 连接里。TCP 看不到 A 和 B，它只看到一串字节流。一旦网络发生丢包（假设丢的是 A 的一个数据包），TCP 为了保证顺序，就会把后续收到的 B 的数据包也扣留在操作系统的缓存里不给浏览器，导致 B 也被无辜卡死。

HTTP/3 (QUIC) 的终极解决

QUIC 在底层原生支持了“流（Stream）”的概念。
在 QUIC 中，请求 A 和请求 B 是相互独立的流。如果 A 的数据包丢了，QUIC 只会暂停 A 的处理等待重传，而毫无关联的 B 的数据包会被直接放行给浏览器。至此，队头阻塞问题被彻底解决。
